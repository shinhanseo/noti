# 개발 계획

## 구현 원칙

- 화면을 한꺼번에 완성하기보다 알림 한 건이 `수집 → 저장 → 판정 → 표시 → 수정`되는 세로 흐름을 먼저 완성한다.
- 각 단계는 빌드, 테스트, 실기기 확인이 끝난 뒤 다음 단계로 넘어간다.
- 체크인/브리핑과 머신러닝은 MVP 완료 전에는 구현하지 않는다.

## 1단계: 기반 확인

- [x] Compose 프로젝트 생성
- [x] 패키지명 `com.hanseo.noti` 설정
- [x] GitHub 저장소 연결
- [x] compileSdk 36.1 호환 의존성으로 빌드 설정 정리
- [ ] `assembleDebug` 성공 확인
- [ ] 기본 앱을 에뮬레이터에서 실행

완료 조건: 깨끗한 체크아웃에서 디버그 APK를 만들 수 있다.

## 2단계: 알림 수집 실험

- [ ] `NotificationListenerService` 선언 및 구현
- [ ] 알림 접근 권한 화면으로 이동하는 온보딩 구현
- [ ] 권한 상태를 앱에서 확인
- [ ] 제목, 본문, 패키지명, 수신 시각을 안전하게 추출
- [ ] 개발 중에는 개인정보가 없는 샘플 또는 마스킹된 로그만 사용
- [ ] 앱 백그라운드, 태스크 제거, 화면 꺼짐 상태에서 실기기 검증

완료 조건: 테스트 알림이 도착하면 앱 화면 상태와 관계없이 정규화된 데이터 한 건을 확인할 수 있다.

## 3단계: 로컬 저장

- [ ] Room 의존성 및 스키마 추가
- [ ] 알림 Entity, DAO, Repository 구현
- [ ] 동일 알림 키의 갱신·삭제 처리
- [ ] 오래된 데이터 정리 정책 구현
- [ ] 알림 데이터의 백업 제외 여부 적용

완료 조건: 앱 프로세스를 종료하고 다시 열어도 수집된 알림이 복구된다.

## 4단계: 중요도 엔진

- [ ] Android에 의존하지 않는 `ImportanceEngine` 구현
- [ ] 중요 앱, 중요 발신자, 포함·제외 키워드 규칙 구현
- [ ] 문맥 신호와 등급 경계값 구현
- [ ] 점수와 판정 이유를 함께 저장
- [ ] 단위 테스트 작성

완료 조건: 고정 테스트 데이터에 대해 같은 점수와 이유를 재현하며, 모든 중요 판정에 설명이 존재한다.

## 5단계: 핵심 화면

- [ ] 알림 접근 권한 온보딩
- [ ] 중요 알림 우선 홈
- [ ] 전체 알림 목록과 필터
- [ ] 판정 이유 바텀시트
- [ ] 빈 상태, 로딩, 권한 해제 상태

완료 조건: 사용자가 수집된 알림과 판정 이유를 끊김 없이 확인할 수 있다.

## 6단계: 사용자 피드백

- [ ] 중요 알림의 `판단이 맞아요`와 `덜 중요해요`
- [ ] 일반 알림의 `중요하게 보기`
- [ ] 잘못 분류된 이유 선택 또는 직접 입력
- [ ] 피드백을 사용자 규칙으로 변환
- [ ] 관련 알림 재판정

완료 조건: 사용자의 한 번의 수정이 이후 유사한 알림의 결과에 일관되게 반영된다.

## 7단계: 안정화 및 포트폴리오 정리

- [ ] 삼성 실기기에서 제조사별 백그라운드 동작 확인
- [ ] 알림 대량 수신과 반복 갱신 성능 확인
- [ ] 민감정보 로그 및 백업 설정 점검
- [ ] 핵심 로직 테스트와 권한 시나리오 테스트 보강
- [ ] README에 실행 화면, 문제 정의, 기술 선택, 트러블슈팅 기록
- [ ] 릴리스 빌드 확인

완료 조건: 주요 사용자 흐름이 실기기에서 재현되고 README만으로 프로젝트의 문제와 기술적 선택을 설명할 수 있다.

## 이후 검토

- 시간대별 체크인 및 일일 브리핑
- 사용자 데이터로 조정되는 경량 온디바이스 분류기
- 알림 요약
- 내보내기·동기화 기능

이 기능들은 MVP 사용자 테스트에서 필요성이 확인된 경우에만 우선순위를 정한다.
