# 중요도 판정 정책

## 목표

MVP는 머신러닝 모델보다 명확한 규칙을 먼저 사용한다. 각 규칙은 점수와 이유를 함께 반환하며, 최종 결과는 사용자가 수정할 수 있어야 한다.

이 문서의 점수는 초기 가설이다. 실제 테스트 데이터와 사용자 피드백을 바탕으로 조정한다.

## 판정 순서

1. 알림을 제목, 본문, 앱, 발신자, 시간 등 공통 형식으로 정규화한다.
2. 사용자 강제 규칙을 확인한다.
3. 제외 신호를 적용한다.
4. 중요 신호를 적용한다.
5. 총점으로 등급을 정하고 판정 이유를 저장한다.
6. 사용자 피드백이 들어오면 규칙을 갱신하고 관련 알림을 재평가한다.

## 강제 규칙

점수보다 사용자의 명시적 설정을 우선한다.

- `항상 중요`: 해당 앱 또는 발신자의 알림을 중요로 분류한다.
- `항상 일반`: 해당 앱 또는 알림 유형을 일반으로 분류한다.
- 사용자가 직접 지정한 제외 키워드는 일반 분류를 강하게 우선한다.
- 서로 충돌하면 더 구체적인 규칙을 우선한다: 키워드/발신자 규칙 → 앱 규칙 → 기본 규칙.

## 초기 점수표

| 신호 | 점수 | 사용자에게 보여줄 이유 예시 |
|---|---:|---|
| 사용자가 중요 앱으로 지정 | +30 | 중요한 앱에서 온 알림이에요 |
| 사용자가 중요 발신자로 지정 | +35 | 중요한 사람이 보낸 알림이에요 |
| 중요 키워드 포함 | +20 | 중요한 단어가 포함되어 있어요 |
| 날짜·시각·마감 표현 포함 | +15 | 일정이나 마감 시간이 함께 있어요 |
| 답변·제출·확인 등 행동 요청 | +15 | 확인하거나 처리할 요청이 있어요 |
| 결제·예약·배송 상태의 중요한 변경 | +15 | 결제·예약·배송 상태가 변경됐어요 |
| 짧은 시간 안에 같은 문맥이 반복됨 | +5 | 관련 알림이 반복해서 도착했어요 |
| 사용자가 제외 키워드로 지정 | -40 | 제외하도록 설정한 표현이 포함됐어요 |
| 광고·혜택·이벤트·추천 신호 | -30 | 홍보성 알림으로 판단했어요 |
| 단순 백그라운드·진행 상태 | -25 | 상태를 알려주는 알림이에요 |
| 내용이 없거나 의미 있는 텍스트 추출 실패 | -10 | 판단할 내용이 충분하지 않아요 |

하나의 문구가 여러 번 등장해도 같은 규칙의 점수는 한 번만 적용한다. `결제`처럼 문맥에 따라 의미가 달라지는 단어는 단독 키워드가 아니라 상태 변화 또는 행동 요청과 함께 평가한다.

## 등급

| 총점 | 등급 | 처리 |
|---:|---|---|
| 40 이상 | 중요 | 홈의 중요 알림 영역에 표시 |
| 20~39 | 확인 필요 | 전체 알림에 표시하고 사용자의 판단을 수집할 후보 |
| 19 이하 | 일반 | 전체 알림에 표시 |

강제 규칙이 존재하면 등급표보다 우선한다. MVP 초기에는 중요한 알림을 놓치는 비용이 더 크므로 테스트 과정에서 경계값을 보수적으로 조정한다.

## 판정 이유 표시

- 내부 점수 전체를 그대로 노출하기보다 상위 1~3개의 구체적인 이유를 보여준다.
- 같은 의미의 이유는 합친다.
- AI가 사용되지 않았는데 `AI가 판단했어요`라고 표시하지 않는다.
- 부정 신호는 일반 알림의 상세 화면에서만 보여주고 사용자를 탓하는 표현을 피한다.

예시:

```text
중요한 앱에서 온 알림이에요
카카오톡을 중요한 앱으로 설정했어요

업무 요청과 마감 시간이 함께 있어요
‘회의 자료’와 ‘오후 3시 전’이 포함됐어요
```

## 피드백 처리

### 중요 알림에서

- `판단이 맞아요`: 현재 판정을 확인한다. 즉시 큰 가중치 변경은 하지 않고 정확도 측정에 사용한다.
- `덜 중요해요`: 앱, 발신자, 키워드 중 어떤 이유가 틀렸는지 선택하게 하고 해당 규칙의 가중치를 낮추거나 제외 규칙을 만든다.

### 일반 알림에서

- `중요하게 보기`: 즉시 중요로 변경한 뒤 이유를 수집한다.
- 이유 후보: 일정·마감, 업무·개인 요청, 결제·금융·배송 상태, 중요한 사람·앱, 직접 입력.
- 선택한 이유를 해당 알림에만 적용할지 이후 유사 알림에도 적용할지 구분한다.

## 향후 개인화

규칙 기반 판정과 피드백 데이터가 충분히 쌓인 뒤, 경량 온디바이스 모델을 보조 점수로 검토한다. 모델이 추가돼도 사용자 강제 규칙과 제외 규칙이 우선하며, 모델 결과 하나만으로 판정을 설명하지 않는다.
